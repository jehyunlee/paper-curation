# M2F: Automated Formalization of Mathematical Literature at Scale

> **저자**: Zichen Wang, Wanli Ma, Zhenyu Ming, Gong Zhang, Kun Yuan, Zaiwen Wen | **날짜**: 2026-02-19 | **Repository**: arXiv | **arXiv ID**: 2602.17016

---

## Essence

수학 교과서와 연구 논문을 Lean 4 형식 증명 라이브러리로 자동 변환하는 최초의 프로젝트 규모(project-scale) 에이전틱 프레임워크 M2F(Math-to-Formal)를 제안한다. 2단계 파이프라인(Stage 1: 명제 컴파일, Stage 2: 증명 복구)과 검증기 인증 수정(VeriRefine) 원칙을 통해, 약 3주 만에 479페이지의 실해석학/볼록해석학 교과서로부터 153,853줄의 Lean 코드(241개 파일, 4,116개 선언)를 생성하여 end-to-end 빌드에 성공하였다. FATE-H 벤치마크에서 96% 증명 성공률을 달성하여 Seed-Prover 1.5(80%)를 16%p 상회하였다.

## Motivation

- **알려진 사실**: 수학의 형식화(formalization)는 기계적 검증을 가능하게 하지만, 개별 정리 수준에 머물러 있으며 프로젝트 규모로의 확장은 거의 다루어지지 않음. mathlib 등 형식 라이브러리의 성장은 전문가 팀의 수개월~수년 노력에 의존
- **격차(Gap)**: 기존 신경 정리 증명기(neural theorem prover)와 에이전틱 시스템은 잘 구성된 명제(well-formed statement) 하에서 개별 증명 목표를 닫는 데 집중하며, 파일 간 의존성 관리, import 해결, 프로젝트 전체 컴파일이라는 핵심 병목을 해결하지 못함
- **접근법**: 문서를 원자적 블록으로 분해 -> 의존성 추론 및 정렬 -> 명제 스켈레톤 생성 및 반복 수정 -> 증명 복구의 2단계 파이프라인으로, Lean 툴체인 피드백이 편집의 수락/거부를 결정하는 VeriRefine 프로토콜을 중심으로 구성

## Achievement

1. **프로젝트 규모 형식화 달성**: 실해석학(312p, 34,327 LoC), 볼록해석학(140p, 105,682 LoC), 연구 논문(27p, 13,844 LoC) 등 총 479페이지를 153,853줄 Lean 라이브러리로 변환, end-to-end 빌드 성공
2. **매칭된 명제 하 100% 증명 성공**: 감사(audit)를 통과한 875개 증명 의무를 모두 자동으로 해결 (PSR=100%)
3. **FATE-H 벤치마크에서 SOTA**: 96% 증명 성공률(완전 자동), 31개 보조정리 맵 추가 시 97%, Seed-Prover 1.5(80%) 대비 +16%p
4. **벤치마크 오류 발견**: FATE-H의 한 문제에서 Lean 명제 자체가 비일관적(inconsistent)임을 검증기 기반 추적으로 발견, GPT-5.2 Pro를 포함한 LLM 기반 의미 검사로는 탐지되지 않은 오류
5. **VeriRefine의 단조성 보장**: 수락된 패치 시퀀스가 단계 목적함수에 대해 단조 감소하여, 진동과 퇴행을 방지

## How

- **Stage 1 (명제 컴파일)**: LaTeX -> JSON 전처리 -> 원자적 블록 분해 -> 의존성 추론 및 정렬 -> LLM이 Lean 스켈레톤 생성(sorry 허용) -> 파일별 VerifyFile 호출 -> 오류 시 범위 지정 패치 제안 -> VeriRefine으로 수락/거부. SCC=100%, ARR<0.5 (대부분 1회 시도로 성공)
- **Stage 2 (증명 복구)**: 컴파일 가능한 P1에서 시작 -> sorry 위치 탐색 -> 목표/컨텍스트 조회 -> 증명 계획 수립 -> 패치 제안 -> VeriRefine으로 수락/거부 -> 실패 시 재계획(replan). 검증기 호출 예산 내에서 반복
- **검증 메트릭**: 파일 수준 오류 수 E(Delta_f), Stage 1에서는 범위 내 오류 수 L, Stage 2에서는 sorry 수 H를 우선순위 순서로 비교
- **인프라**: Lean 4 v4.26.0-rc2, Codex CLI(reasoning effort="high"), LSP 기반 MCP 서버, Intel Xeon 6982P-C 16코어/128GiB RAM
- **검증**: 명제 충실성은 도메인 전문가의 수동 감사(provenance-linked manual audit)로 보장

## Originality

교과서/논문 규모의 수학 형식화를 "프로젝트 컴파일" 문제로 정식화하고, 검증기 인증 수정(VeriRefine)이라는 단조성 보장 원칙으로 장기 실행의 안정성을 확보한 최초의 시스템이다. 개별 정리 증명이 아닌 프로젝트 전체의 import/namespace/의존성 안정화를 핵심 과제로 설정한 관점 전환이 독창적이며, 검증기 피드백만으로 편집 수락을 결정하는 설계가 LLM 기반 제안과 형식 검증을 깔끔하게 분리한다. FATE-H 벤치마크 오류를 검증기 워크플로우로 발견한 사례는 형식 검증의 부가 가치를 잘 보여준다.

## Limitation & Further Study

### 저자들이 언급한 한계
- 남은 병목은 자연어 그라운딩(natural-language grounding)과 라이브러리 탐색(library navigation)
- FATE-H에서 4개 미해결 문제 중 2개는 전문가 보조정리 맵 없이 자동 해결 불가
- 명제 충실성 검증이 수동 감사에 의존하여 확장성에 한계

### 리뷰어 판단 아쉬운 점
- LLM 백엔드(Codex CLI)의 구체적 모델 버전과 비용이 명시되지 않아 재현성 판단이 어려움
- Stage 2에서 "매칭된 명제(matched statements)" 조건이 전제되므로, Stage 1에서 생성된 명제 자체의 의미적 정확성은 별도 문제로 남음
- 볼록해석학의 105,682줄이 실해석학의 34,327줄 대비 과도하게 많은데, 코드 중복이나 불필요한 보조 정리 생성 가능성에 대한 분석이 부재
- 정규 도메인(일반 위상, 대수, 조합론 등)으로의 일반화 가능성이 실증되지 않음
- 전체 파이프라인의 end-to-end 실행 시간("약 3주")에 대한 세부 분석(LLM 호출 수, 비용 등)이 본문에서 부족

### 후속 연구
- 자동화된 명제 충실성 검증 방법론 개발 (수동 감사 대체)
- 대수학, 위상수학, 조합론 등 다른 수학 분야로의 확장
- 오픈소스 LLM 백엔드로의 이식성 검증
- mathlib에 직접 기여 가능한 코드 품질 달성을 위한 스타일 준수 및 중복 제거 자동화
