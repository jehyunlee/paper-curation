Essence (본질)

자연과학 분야(수학, 물리, 화학, 생물, 재료과학)의 실제 연구 코딩 문제를 과학자들이 직접 큐레이션한 벤치마크 SciCode를 제안한다. 80개 주요 문제(main problem)가 338개 하위 문제(subproblem)로 분해되며, 각 문제는 과학적 배경지식 회상, 추론, 코드 합성을 동시에 요구한다. 가장 현실적인 평가 설정에서 최고 성능 모델 Claude3.5-Sonnet이 주요 문제의 4.6%만 해결할 수 있어, 현재 LM의 과학 코딩 능력의 한계를 명확히 보여준다.

Figure 3

Motivation (동기)

기존 코딩 벤치마크(HumanEval, MBPP 등)가 LM 성능 향상에 따라 포화되고 있으며, 합성적으로 난이도를 높인 벤치마크는 실세계 응용과의 괴리가 있다. 한편 과학 연구에서 코드 개발은 핵심 활동이지만, 과학자들의 내부 코드는 거의 공개되지 않아 LM 훈련 데이터에 노출이 적다. 이러한 데이터 오염 방지와 실세계 과학 연구 적용성을 동시에 확보할 수 있는 도전적 벤치마크가 필요했다.

Achievement (성과)

  • 벤치마크 구축: 16개 세부 분야에서 80개 주요 문제 / 338개 하위 문제, 노벨상 수상 연구 관련 문제 다수 포함
  • 표준 설정(배경지식 없이 + 생성된 이전 해답 사용):
  • Claude3.5-Sonnet: 하위 문제 26.0%, 주요 문제 4.6%
  • GPT-4o: 하위 문제 25.0%, 주요 문제 1.5%
  • 오픈 모델 최고(Deepseek-Coder-v2): 주요 문제 3.1%
  • 배경지식 제공 시: 모든 모델이 상당히 향상 (Claude3.5-Sonnet: 12.3%), 과학 지식 부족이 주요 병목임을 확인
  • Gold 하위 해답 조건부: 이전 단계의 gold 해답이 많아질수록 성능 향상, 단 9개 이상에서는 긴 컨텍스트 관리 어려움으로 하락
  • 3단계 검증(동분야 과학자, 타분야 과학자, GPT-4)으로 높은 품질 보장

How (방법론)

  1. 문제 설계: 과학자들의 일상 연구 워크플로(수치 방법, 시스템 시뮬레이션, 과학 계산)에서 문제 추출
  2. 구조화된 분해: 각 주요 문제를 복수의 하위 문제로 분해, 각 하위 문제는 독립적 Python 함수 구현 + 통합
  3. 평가 설계:
  4. 수치 테스트(입출력 쌍 비교) + 도메인 특화 테스트(논문 결과 재현, 해석해 검증)
  5. 4가지 평가 모드: 배경지식 유/무 x Gold/생성 이전 해답
  6. 3라운드 검증: 동분야 과학자 교차검증 -> 타분야 과학자 명확성 검증 -> GPT-4 기반 모호성 제거
  7. 평가: 10개 SOTA 모델(상용 6개 + 오픈 4개)에 대해 zero-shot 프롬프트로 pass@1 측정

Originality (독창성)

  • 과학자 큐레이션: PhD 이상 연구자가 직접 문제 설계, 솔루션 작성, 검증한 최초의 대규모 과학 코딩 벤치마크
  • 다단계 분해 구조: 단순 함수 완성이 아닌, 복수 하위 함수를 구현하고 통합하는 과학적 워크플로 반영
  • 데이터 오염 최소화: 내부 연구 코드 기반으로 공개 데이터셋과 중복 없음
  • 노벨상 연구 포함: DFT(1998), 중성미자 진동(2015), Haldane 모델(2016), 광학 집게(2018), 스핀 글래스(2021) 등
  • 유연한 평가 설정: 배경지식 토글, gold/생성 해답 전환 등으로 다양한 모델 능력 평가 가능

Limitation & Further Study (한계 및 후속 연구)

  • 규모 한계: 80개 주요 문제는 통계적으로 신뢰성 있는 분야별 비교에 부족할 수 있음
  • 물리 편향: 전체 문제의 46%가 물리학에 편중, 화학/생물/재료과학 비중이 상대적으로 적음
  • Python 전용: 실제 과학 연구에서 사용되는 Fortran, C++, Julia, MATLAB 등 미포함
  • 단일 생성(pass@1)만 평가: pass@k, 자기 수정, 에이전트 기반 접근 등 다양한 전략 미탐색
  • 테스트 케이스의 포괄성: 수치적 허용 오차 설정, edge case 커버리지에 대한 체계적 분석 부족
  • 정적 벤치마크: SWE-bench와 달리 자동 확장 메커니즘 없음
  • 후속 연구: 실험 데이터 분석, 시각화, 도메인 특화 라이브러리 활용 등으로 확장 가능
