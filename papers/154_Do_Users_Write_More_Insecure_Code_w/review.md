# Do Users Write More Insecure Code with AI Assistants?

## 메타 정보
- **저자**: Neil Perry, Megha Srivastava, Deepak Kumar, Dan Boneh
- **출판**: ACM CCS 2023
- **DOI**: [10.1145/3576915.3623157](https://doi.org/10.1145/3576915.3623157)

---

## Essence (본질)
AI 코드 어시스턴트(OpenAI Codex 기반)를 사용하는 개발자가 보안 관련 프로그래밍 과제에서 **더 취약한 코드를 작성하는지**를 측정한 최초의 대규모 사용자 연구이다. 47명의 참가자가 Python, JavaScript, C로 5개 보안 관련 과제를 수행했으며, AI 어시스턴트에 접근할 수 있었던 실험군이 대조군보다 유의미하게 덜 안전한 코드를 작성했고, 동시에 자신의 코드가 안전하다고 더 강하게 믿는 **과신(overconfidence)** 경향을 보였다.

## Motivation (동기)
GitHub Copilot 같은 AI 코드 어시스턴트가 생산성을 향상시킬 수 있다는 연구는 있었지만, 실제 사용자가 AI와 상호작용하는 맥락에서 보안 취약점이 어떻게 도입되는지에 대한 체계적 연구는 부재했다. 기존 연구(Pearce et al., 2022)는 합성 프롬프트에 한정되어 실제 개발자 행동을 반영하지 못했고, 유일한 사용자 연구(Sandoval et al., 2022)는 C 언어에 한정되어 결론이 불분명했다. 다양한 언어와 보안 영역을 아우르는 통제된 실험이 필요했다.

## Achievement (성과)
- **핵심 발견 (RQ1)**: AI 어시스턴트 접근 가능 그룹이 5개 과제 중 4개에서 유의미하게 덜 안전한 코드 작성
  - Q1(암호화): 실험군 insecure 비율 유의미하게 높음 (p=0.017)
  - Q2(서명): 실험군 secure 3% vs 대조군 21% (p=0.039)
  - Q3(샌드박스): 실험군 secure 12% vs 대조군 29%, symlink 미처리 (p=0.019)
  - Q4(SQL): SQL 인젝션 취약 36% vs 7% (p=0.041)
  - Q5(C 문자열): 혼합 결과 (유일한 예외)
- **핵심 발견 (RQ2)**: AI 접근 가능 참가자가 모든 문제에서 자기 코드의 보안성을 더 높게 평가 -- 실제 보안 수준과 반비례하는 과신 현상
- **핵심 발견 (RQ3)**: AI를 덜 신뢰하고 프롬프트를 적극적으로 수정(temperature 조정, 재구성)한 참가자가 더 안전한 코드 작성
- AI 출력을 반복적으로 다음 프롬프트로 사용하는 "순환(cycle)" 패턴이 취약점 증폭 유발
- 87%의 안전한 응답이 AI 출력에서 상당한 수정을 거침

## How (방법론)
1. **실험 설계**: 47명 참가자(학부생 66%, 대학원생 19%, 전문가 15%)를 실험군(33명, AI 접근 가능)과 대조군(14명)으로 무작위 배정 (2:1 비율)
2. **과제**: 5개 보안 관련 프로그래밍 과제 (암호화/복호화, ECDSA 서명, 샌드박스 디렉토리, SQL 삽입, C 문자열 처리)
3. **도구**: 커스텀 UI(React/Redux/Electron 기반)로 코드 작성, AI 쿼리, 모든 상호작용 자동 기록 + 화면/음성 녹화
4. **분석**:
   - 2명의 연구자가 독립적으로 정확성/보안성 코딩 (Cohen-Kappa 0.68~0.97)
   - 로지스틱 회귀로 AI 접근 여부의 효과 모델링 (보안 배경, 프로그래밍 경험, 학생 여부 통제)
   - Benjamini-Hochberg 보정으로 다중 검정 교정
   - 프롬프트 언어/매개변수/수리 전략에 대한 정성적 분석

## Originality (독창성)
- AI 코드 어시스턴트의 보안 영향을 **사용자 행동** 맥락에서 측정한 최초의 포괄적 연구
- 다중 프로그래밍 언어(Python, JavaScript, C)와 다양한 보안 영역(암호화, 입력 검증, 메모리 관리 등)을 아우름
- 프롬프트 전략의 **분류 체계**(Specification, Instruction, Function Declaration, Helper 등)와 **수리 전략**(Retry, Expand Scope, Reword, Change Type 등) 제시
- 보안과 신뢰 간의 **역상관 관계** 발견: 안전한 코드를 작성한 참가자가 AI를 덜 신뢰
- UI 인프라와 익명화 데이터를 공개하여 후속 연구의 재현 가능성 확보

## Limitation & Further Study (한계 및 후속 연구)
- **표본 편향**: 참가자 대부분이 대학생(87%가 18-24세)으로, 실무 소프트웨어 개발자를 대표하지 못함
- **표본 크기**: 47명(실험군 33, 대조군 14)으로 미묘한 효과 감지에 한계 -- 일부 결과가 통계적으로 유의미하지 않음
- **모델 한정**: codex-davinci-002만 사용 -- 이후 GPT-4, Claude 등 더 강력한 모델에서 결과가 달라질 수 있음
- **인위적 환경**: 시간 제한, 성과와 직업적 결과 무관 등 실제 업무 환경과 차이
- **C 언어 과제(Q5)**: AI가 C 코드 생성에 어려움을 겪어 결과 해석이 복잡
- 후속 연구: ChatGPT/Copilot 등 최신 모델에서의 반복, 실무 개발자 대상 연구, 보안 피드백 메커니즘 설계, RLHF 기반 보안 지향 강화학습
